Как работаем:

  1. Нарушать правила можно только при наличии разумной причины.
  2. Стараемся советоваться с напарниками по любому значимому поводу. Чем больше мы знаем о том, что и как делают другие, тем меньше вероятности, что будут какие-то проблемы.
  3. Используем БЕМ в нотации .block__element--modificator
  4. Инструкция по запуску сборочного проекта находится в README.md.
  5. Любые библиотеки стараемся тянуть через bower и подключать из папки /bower_components/, не забывая регистрировать в bower.json. Почему не npm? Потому что тяжело потом искать что-то среди сотни зависимостей. Не забываем, что любой проект из github можно забрать к себе. Например, bower install https://github.com/nicolas-cusan/destyle.css.git#1.0.5 --save. Для dev-зависимостей используем npm.
  6. Если вендорный файл требует изменений или нет возможности подключить через bower, копируем непосредственно в /src/css/vendor/ или /src/js/vendor. Измененные файлы при этом помечаем постфиксом --fixed.
  7. Modernizr по умолчанию подключен. Задача по сборке запускается только при первом билде. Если нужно добавить проверку новых фич, делать это в конфиге из Gruntfile.js.
  8. Скрипты оформляем в отдельные файлы (модули) и складываем в /src/js/modules/. Называем в соответствии с блоками css: header.js, nav-top.js. Модуль имеет право создавать только одну переменную в глобальном пространстве: Header, NavTop.
  9. Работаем через git. Все делаем в одной ветке. Локально можно использовать сколько угодно веток. Merge делаем через rebase. Это значит, что вы переносите свои коммиты поверх чужих так, будто вы их делали позже по времени. Цель -- упростить восприятие истории изменений. Пример очень нежелательной ситуации на втором рисунке.
